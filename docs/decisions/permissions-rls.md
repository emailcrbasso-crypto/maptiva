# Permissions & RLS Matrix- Status: Draft- Date: 2026-05-07- Owner: Rafael Medeiros- Related ADRs:  - docs/decisions/0001-people-vs-cycle-participants.md  - docs/decisions/0002-rls-strategy.md## GoalDefine the database access matrix for the 360 assessment platform using explicit Supabase/Postgres RLS policies per table, role, and operation.## Principles- Enforce tenant isolation in the database.- Prefer explicit policies over broad shared policies.- Separate read and write permissions.- Keep public response submission out of direct table access.- Use backend/service-role access only for controlled internal jobs.- Keep authorization logic auditable and testable.## Role model- **owner**: tenant owner, full tenant-scoped control.- **admin**: tenant administrator, broad tenant-scoped control.- **manager**: manages cycles and can read reporting for managed scope.- **collaborator**: limited operational access.- **participant**: cycle-bound user with limited self-scope access.- **anonymous_magic_token**: no direct table access; only Edge Function submission flow.- **service_role**: backend-only privileged access for imports, reminders, and internal jobs.## Access helpersRecommended helper functions:- `is_tenant_member(target_tenant_id uuid)`- `has_tenant_role(target_tenant_id uuid, roles text[])`- `is_cycle_manager(target_cycle_id uuid)`- `is_cycle_participant(target_cycle_id uuid)`- `is_submission_token_valid(target_evaluator_id uuid, token uuid)`- `is_service_context()`These helpers should be `stable`, small, and index-friendly.## Table matrix| Table | SELECT | INSERT | UPDATE | DELETE | Notes ||---|---|---|---|---|---|| `tenants` | owner/admin only | service role only | owner/admin only | owner/admin only | Tenant root record. || `tenant_memberships` | owner/admin only | owner/admin only | owner/admin only | owner/admin only | Membership administration only. || `people` | tenant members in scope | owner/admin only | owner/admin/manager for scoped edits | owner/admin only | Reusable identity table. || `cycle_participants` | tenant members in scope; General authenticated users should not list tenants outside their membership scope.### tenant_membershipsMembership rows are sensitive and should only be visible to tenant owners and admins. Creation and removal of memberships should also be restricted to owner/admin flows.### people`people` represents reusable tenant-scoped identity. Read access follows tenant membership, while writes should be limited to owner/admin and possibly manager-scoped imports or controlled edits.### cycle_participants`cycle_participants` is the cycle-specific snapshot. Managers may read scoped rows for cycles they manage, but mutations should be limited to owner/admin or tightly controlled setup flows.### templatesTemplates are administrative content. Only owner/admin should create or edit them, while tenant members may read active templates if required by product behavior.### cyclesCycle creation and status changes should be owner/admin controlled. Managers may read and operate only on cycles they manage, including progress dashboards and reports.### assignments / evaluatorsThis table maps who evaluates whom and carries the magic token. It must be readable only in the proper tenant and scope, and it should not expose token data to general clients unless specifically required by the Edge Function flow.### responsesDirect writes from the browser should be disabled. Responses are created or updated through the `submit_response` Edge Function using backend privileges after validating the magic token and submission state.### commentsComments follow the same security model as responses. They are persisted only through controlled backend flow, not by direct anonymous table writes.### analytics tablesDerived analytics must be readable only within the tenant scope and generally written by service-role jobs or controlled backend processes.### exports and auditExport records and audit logs should be owner/admin only for reads, with writes restricted to backend jobs.## Public submission flowThe public responder must never receive direct broad table access. Instead:1. The user opens a magic-token URL.2. The Edge Function validates token existence, expiry, and completion state.3. The Edge Function writes responses/comments using privileged backend access.4. The token is invalidated or marked completed after successful submission.## Policy conventions- Use `USING` for row visibility.- Use `WITH CHECK` for row creation and updates.- Avoid policy recursion and complex subqueries when a helper function can express the rule cleanly.- Prefer separate policies for `SELECT`, `INSERT`, `UPDATE`, and `DELETE`.- Keep backend-only actions documented as exceptions.## Validation checklistBefore merge, verify that:- all exposed tables have RLS enabled,- no table allows cross-tenant reads,- no client path can insert responses directly,- the magic-token submission works only through the Edge Function,- and helper functions are indexed and tested.## NotesThis document is the source of truth for policy intent. The SQL migrations must implement the same matrix, and any changes to role behavior or submission flow require updating this file and the relevant ADR.
