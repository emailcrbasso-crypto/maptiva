# ADR 0001 — Adopt `people` + `cycle_participants` instead of a single `participants` table- Status: Accepted- Date: 2026-05-07- Decision makers: Rafael Medeiros- Consulted: Claude Code / product architecture review- Related docs:  - docs/modules/templates.md  - docs/modules/cycles.md  - docs/modules/assignments.md  - docs/modules/responses.md  - docs/modules/scoring-analytics.md  - docs/modules/reports-exports.md  - docs/modules/participants.md  - docs/permissions-rls.md  - docs/integrations/maptiva-grid.md## Context and problem statementThe Maptiva platform needs to support longitudinal analysis, reusable people records, and cycle-specific snapshots for assessment, scoring, reporting, and Grid integration. A single `participants` table tied directly to a cycle makes it harder to preserve history across multiple cycles, compare results over time, and evolve the domain cleanly.## Decision drivers- Reuse the same person across many cycles without duplicating identity data.- Preserve cycle-specific state, metadata, and historical context.- Support longitudinal reporting and future Grid / Nine Box use cases.- Keep the schema consistent with multi-tenant architecture and RLS.- Avoid coupling the master entity to a specific assessment round.## Considered options### Option 1 — Single `participants` table linked to cycleStore participants directly on the cycle record or via a cycle-bound participants table only.### Option 2 — Split model: `people` + `cycle_participants`Store reusable person identity in `people`, and create a cycle-specific snapshot in `cycle_participants`.## Decision outcomeChosen option: **Split model: `people` + `cycle_participants`**.This option best supports reusable identity, cycle snapshots, historical integrity, and downstream analytics needs.## Consequences### Good- People can be reused across multiple cycles without duplicating the master identity.- Cycle-specific fields like manager, metadata, and participation state can be stored per round.- Historical comparisons become more reliable.- Grid integration becomes cleaner because the model naturally separates master identity from cycle context.- The schema is easier to extend for future features such as versioned org structure or longitudinal reporting.### Bad- The schema becomes slightly more complex than a single-table approach.- Import logic must handle both person creation/update and cycle enrollment.- Queries need to join `people` and `cycle_participants` when reading cycle context.## ConfirmationThis decision is confirmed when:- the schema includes `people` as the reusable tenant-scoped entity,- the schema includes `cycle_participants` as the cycle snapshot entity,- assignments and analytics reference `cycle_participants`,- the implementation preserves historical integrity across cycles,- and RLS policies enforce tenant isolation on both tables.## NotesThis ADR supersedes any earlier assumption that a single `participants` table should carry both reusable identity and cycle-specific state. Future architectural changes must be recorded in a new ADR rather than editing this one.
