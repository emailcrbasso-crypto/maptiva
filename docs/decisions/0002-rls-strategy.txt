# ADR 0002 — RLS strategy by table, role, and operation- Status: Accepted- Date: 2026-05-07- Decision makers: Rafael Medeiros- Consulted: Claude Code / product architecture review- Related docs:  - docs/decisions/0001-people-vs-cycle-participants.md  - docs/permissions-rls.md  - docs/modules/templates.md  - docs/modules/cycles.md  - docs/modules/assignments.md  - docs/modules/responses.md  - docs/modules/scoring-analytics.md  - docs/modules/reports-exports.md  - docs/integrations/magic-token-submit-response.md## Context and problem statementThe platform is a multi-tenant 360 assessment system with sensitive cycle data, evaluator assignments, answers, comments, and reporting outputs. Access must be isolated by tenant and by role, while the public answer flow must remain safe for magic-link based submission.## Decision drivers- Prevent cross-tenant data exposure.- Keep database authorization deterministic and auditable.- Support admin, manager, participant, and anonymous magic-link flows.- Avoid relying on client-side filtering for security.- Preserve a clean separation between authenticated app access and public response submission.## Considered options### Option 1 — Broad permissive RLSUse a small number of generic policies across tables.### Option 2 — Explicit RLS per table, role, and operationDefine policies with clear scope for each table and action.## Decision outcomeChosen option: **Explicit RLS per table, role, and operation**.Every exposed table will have RLS enabled, and policies will be written to match the exact access pattern of each role and module.## Policy principles- Enforce tenant isolation at the database level.- Grant the minimum access needed for each role.- Separate read and write behavior.- Treat backend-only and service-role operations as exceptions.- Keep policies simple enough to review and test.- Use helper functions for repeated membership and ownership checks.## Role modelThe initial authorization model assumes these access paths:- **Owner/Admin**: full tenant-scoped access.- **Manager/Lead**: access to cycles and reporting data in their scope.- **Participant/Assessor**: access only to their own assigned evaluation context.- **Anonymous respondent via magic token**: submit only through Edge Function.- **Service role / backend**: limited to server-side jobs, imports, reminders, and report generation.## ScopeThis ADR applies to the exposed schema, especially:- tenants- tenant_memberships- people- cycle_participants- templates- cycles- assignments or evaluators- responses- comments- analytics tables- exports and audit tables## Implementation approach1. Enable RLS on every table exposed to application access.2. Write policies per table and per operation instead of shared permissive rules.3. Use helper functions for tenant membership, role checks, and managed-cycle checks.4. Keep direct public writes disabled for response submission.5. Implement `submit_response` as an Edge Function that validates the magic token and writes with privileged backend access.6. Document every exception explicitly in the permissions spec.## Consequences### Good- Strong protection against tenant data leakage.- Easier auditing of who can read or mutate each table.- Better alignment with the magic-link submission flow.- Clearer behavior when new modules are added.- Lower risk of accidental overexposure in client code.### Bad- More SQL policies to maintain.- More work when adding new tables or roles.- Need for tests covering each access path.- Some flows require backend orchestration instead of direct client writes.## ConfirmationThis decision is confirmed when:- all exposed tables have RLS enabled,- policies exist for each relevant role and operation,- tenant membership is checked consistently,- magic-link submission works only through the Edge Function,- and backend-only jobs are documented as exceptions.## NotesThis ADR is intentionally narrower than the permissions specification. The exact SQL policies will live in `docs/permissions-rls.md` and in the migration files. Any future change to access semantics must be captured in a new ADR.
